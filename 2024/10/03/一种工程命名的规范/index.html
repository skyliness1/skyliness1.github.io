一种工程命名的规范

Thallium Acid2024-10-032024-10-03

目的：

  1. 多人代码合作需要互相读代码，减轻代码阅读难度。
  2. 更容易互相调用别人写的模块。
  3. 项目需要快速推进，但不能以质量换速度，屎山会使得后期修改和重构工作量提升，在代码整洁上花部分时间目的是提高总体的工作效率。

命名规范

​ 类、属性、字段和函数的命名一定要准确，减少歧义避免误导，不要依赖注释解释函数功能，要让别人看到名字就明白具体功能。

​ 命名不用担心标识符的长度，准确清晰为第一要务（一些大家认同的单词简写可以接受，如pos、cur等）。

​ 类、属性和字段用名词，函数、方法用动词，接口使用形容词或名词（如IEnumerable），返回bool类型的方法、属性、字段以is或has等判断句意单词开头。

​ 所有命名使用camel Case（不带下划线，单词以大写字母隔开）。

​ 字段访问修饰符只用Private，用于类的内部使用，所有需要暴露给外部的字段使用一个Public属性包裹（这时候需要序列化需要用到OdinSerializer），private修饰符不要因为是默认修饰符就不写。

​ 属性非必要时使用自动属性，需要使用私有字段时注意命名一致。

  • 接口：以大写I开头
  • 常量、只读字段：全大写
  • 类、结构体、枚举类型：以大写字母开头
  • 事件：以小写on开头 onBefore/onAfter…
  • 字段：以m_加小写字母开头
  • 属性：以小写字母开头
  • 方法：以大写字母开头

垂直格式组织

​ 为了方便理解某一个类（或者接口、结构体）的功能，类内部的垂直格式需要固定，方便快速找到想要找到代码段的位置。一个类内部的组织顺序如下：

  • 构造器
  • 解构器
  • static public属性（一般都会是是单例）
  • public属性
  • protected属性
  • private属性
  • public事件
  • private事件
  • private字段
  • static方法
  • virtual方法
  • public方法
  • protected方法
  • private方法
  • Unity的事件方法（Awake、Start等），按照事件的发生时间排序
  • 类内的嵌套定义（Enum、结构体、类中类）

Monobehaviour代码初始化

  • Awake()：初始化代码内部字段
  • Start()：初始化代码对外部对象的引用
  • OnEnable()：事件订阅，初始化代码对外部对象的引用
  • OnDisable()：事件取消订阅

类和函数的体量

​ 类和函数的功能尽量短小精悍，避免出现很长的代码或很长的函数，当类开始变得臃肿，需要考虑拆解类的功能，将一个类拆成多个，把每个类功能做的尽量单一而准确。函数尽量制作“一件事”，很多事发生在一个过程中时，将他们分别用一个私有函数表示并拆开。

场景管理

​ 所有人在测试时使用个人专属的场景，尽量避免场景冲突，与正式版本相关的场景由主程管理。

工作流程和版本管理

分支结构如下：

main

└ devMain

└ 个人1分支

│ └ 个人1dev分支

└ 个人2分支

│ └ 个人2dev分支

…

​ 主分支main是整个项目的版本，用来记录发布用版本；devMain是个人开发分支的合并版本，这两个分支都有主程负责合并和管理。个人开发时在自己的dev分支上开发，在个人dev分支上时可以随时提交，当完成某一大功能并自我代码审查确认无误后再向个人分支合并，并由主程在个人分支上进行代码审查。

​ 代码审查会在Plastic SCM上进行。

代码规范

  • 属性
    • 无论权限
    • 小写单词
  • 字段
    • 只能私有
    • 如果需要公开，就添加属性
    • m_开头
  • 特性使用
    • OdinSerialize
      • 不适用
    • 需要序列化的字段
      • 使用[SerializeField]
  • 整洁书写效果
    • 排列顺序
      • 公开属性
      • 私有属性
      • 序列化私有字段
      • 私有字段
      • 函数
        • 公开函数
          • 公开函数调用的私有函数尽量放公开函数的下面
        • 私有函数
        • 按照生命周期顺序排列的函数
